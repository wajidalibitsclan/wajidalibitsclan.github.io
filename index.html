PREREQUISITES

Find Your Username

Before starting, find your system username. It appears in all file paths throughout this guide.

Windows

Open Command Prompt or PowerShell:

Command Prompt / PowerShell
$ whoami
DESKTOP-ABC123\youruser

$ echo %USERNAME%
youruser

SSH keys stored at: C:\Users\youruser\.ssh\

Ubuntu / Linux

Terminal
$ whoami
youruser

$ echo $USER
youruser

SSH keys stored at: /home/youruser/.ssh/

ℹ️
NOTE — Replace youruser with your actual username throughout this entire guide.
PART 1

Generate SSH Key

Open Your Terminal

Windows: Win + R → type cmd → Enter, or search Git Bash

Ubuntu: Ctrl + Alt + T to open terminal

Generate Key Pair

Run on both OS:

Terminal
$ ssh-keygen -t ed25519 -C "youruser@example.com"

Windows output

Generating public/private ed25519 key pair.
Enter file in which to save the key (C:\Users\youruser\.ssh\id_ed25519):

Ubuntu output

Generating public/private ed25519 key pair.
Enter file in which to save the key (/home/youruser/.ssh/id_ed25519):

Press Enter to accept the default.

Set Passphrase (Optional)

Enter passphrase (empty for no passphrase):
Enter same passphrase again:

Type a strong passphrase or press Enter to skip.

Verify Key Created

Windows

Command Prompt
$ dir C:\Users\youruser\.ssh\

Ubuntu

Terminal
$ ls -la ~/.ssh/

id_ed25519 — Private key (never share)  |  id_ed25519.pub — Public key (goes on server)

View Public Key

Windows

Command Prompt
$ type C:\Users\youruser\.ssh\id_ed25519.pub
✨
TIP — Copy to clipboard: clip < C:\Users\youruser\.ssh\id_ed25519.pub

Ubuntu

Terminal
$ cat ~/.ssh/id_ed25519.pub

Copy the entire output — you'll need it for both GitHub and your VPS.

PART 2 — ADDED

Add SSH Key to GitHub NEW

Without this step, your VPS won't be able to git clone or git pull private repositories. Even for public repos, SSH access avoids rate limits and authentication prompts.

Copy Your Public Key

If you haven't already, copy the output from Step 5 above.

Add to GitHub

Go to GitHub.com → click your avatar → Settings → SSH and GPG keys → New SSH key

FieldValue
TitleA descriptive name, e.g. My Laptop or Work PC
Key typeAuthentication Key
KeyPaste entire public key (starts with ssh-ed25519)

Click Add SSH key and confirm with your GitHub password.

Test GitHub Connection

Terminal
$ ssh -T git@github.com

Expected output:

Hi youruser! You've successfully authenticated, but GitHub does
not provide shell access.
⚠️
FIRST TIME? — If asked "Are you sure you want to continue connecting?", type yes and press Enter. This adds GitHub to your known hosts.

Also Add a Key on Your VPS

Your VPS also needs to authenticate with GitHub to run git pull. SSH into your VPS and generate a separate key pair there:

VPS Server
$ ssh-keygen -t ed25519 -C "vps-server"
$ cat ~/.ssh/id_ed25519.pub

Add this VPS public key to GitHub the same way (Settings → SSH keys). Name it something like VPS Production Server.

Test from VPS:

VPS Server
$ ssh -T git@github.com
🚨
CRITICAL — Without this, every git pull in your CI/CD deploy script will fail with "Permission denied (publickey)" on private repos.
PART 3

Copy Public Key to VPS

Method A: ssh-copy-id (Easiest)

Ubuntu / Git Bash
$ ssh-copy-id youruser@your_server_ip

Asks for server password one last time, then installs the key.

Method B: Manual Copy

Terminal
$ ssh youruser@your_server_ip
$ mkdir -p ~/.ssh
$ chmod 700 ~/.ssh
$ echo "PASTE_YOUR_PUBLIC_KEY_HERE" >> ~/.ssh/authorized_keys
$ chmod 600 ~/.ssh/authorized_keys
$ exit
✨
TIP — Replace PASTE_YOUR_PUBLIC_KEY_HERE with output from Step 5.

Windows PowerShell one-liner

PowerShell
$ type C:\Users\youruser\.ssh\id_ed25519.pub | ssh youruser@your_server_ip "mkdir -p ~/.ssh && cat >> ~/.ssh/authorized_keys && chmod 600 ~/.ssh/authorized_keys"

PART 5

Secure the Server

Run these commands on your VPS server.

Create a Non-Root User NEW

If your VPS provider gave you only root access, create a dedicated deploy user first:

VPS Server (as root)
# adduser youruser
# usermod -aG sudo youruser
# rsync --archive --chown=youruser:youruser ~/.ssh /home/youruser
# exit

Now SSH in as the new user: ssh youruser@your_server_ip

Update System

VPS Server
$ sudo apt update && sudo apt upgrade -y

Add Swap Space NEW

Small VPS instances (1–2 GB RAM) can run out of memory during npm run build or composer install. Adding swap prevents crashes:

VPS Server
$ sudo fallocate -l 2G /swapfile
$ sudo chmod 600 /swapfile
$ sudo mkswap /swapfile
$ sudo swapon /swapfile
$ echo '/swapfile none swap sw 0 0' | sudo tee -a /etc/fstab

Verify with free -h — you should see 2 GB swap.

Disable Password Authentication

VPS Server
$ sudo nano /etc/ssh/sshd_config
/etc/ssh/sshd_config
PasswordAuthentication no
PubkeyAuthentication yes
PermitRootLogin no

Restart SSH

VPS Server
$ sudo systemctl restart sshd
⚠️
WARNING — Keep your current session open. Test from a NEW terminal first!

Firewall

VPS Server
$ sudo ufw allow OpenSSH
$ sudo ufw enable
$ sudo ufw status
Status: active
To           Action   From
OpenSSH      ALLOW    Anywhere
PART 6

CI/CD Pipeline Setup

Using GitHub Actions — every push to main triggers automatic testing and deployment to your VPS.

Generate Deploy Key

Ubuntu

Terminal
$ ssh-keygen -t ed25519 -C "github-deploy" -f ~/.ssh/deploy_key

Windows

Command Prompt
$ ssh-keygen -t ed25519 -C "github-deploy" -f C:\Users\youruser\.ssh\deploy_key

Press Enter for no passphrase (required for automation).

deploy_key → Private key → GitHub Secrets  |  deploy_key.pub → Public key → VPS server

🚨
IMPORTANT — Never commit the private key to your repository.
💡
UNDERSTANDING THE KEY FLOW — This deploy key lets GitHub Actions SSH into your VPS. It is separate from the VPS's own SSH key (Part 2 Step 4) which lets your VPS pull code from GitHub. You need both.

Add Public Key to VPS

Terminal
$ ssh youruser@your_server_ip
$ echo "PASTE_DEPLOY_KEY_PUB" >> ~/.ssh/authorized_keys
$ exit

Copy Private Key

Ubuntu

Terminal
$ cat ~/.ssh/deploy_key

Windows

Command Prompt
$ type C:\Users\youruser\.ssh\deploy_key
⚠️
WARNING — Copy the ENTIRE output including BEGIN/END lines.

Add GitHub Secrets

Go to your repo → Settings → Secrets and variables → Actions:

Secret NameValue
VPS_SSH_KEYEntire private key (with BEGIN/END lines)
VPS_HOSTServer IP, e.g. 203.0.113.50
VPS_USERServer username, e.g. youruser
VPS_PORTSSH port, usually 22
ℹ️
NOTE — Secrets are encrypted and never visible in workflow logs.

Create Workflow Directory

Ubuntu
$ cd ~/your-repo
$ mkdir -p .github/workflows
Windows
$ cd C:\Users\youruser\your-repo
$ mkdir .github\workflows

Create Workflow File

Choose the pipeline matching your framework below.

CI/CD PIPELINES

Python Pipelines

⚡

FastAPI

deploy.yml
.github/workflows/deploy.yml
name: Deploy FastAPI to VPS

on:
  push:
    branches: [ main ]

jobs:
  test:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-python@v5
        with:
          python-version: '3.12'
          cache: 'pip'
      - run: |
          pip install --upgrade pip
          pip install -r requirements.txt
      - run: pip install ruff && ruff check .
      - run: python -m pytest tests/ -v

  deploy:
    needs: test
    runs-on: ubuntu-latest
    if: github.ref == 'refs/heads/main'
    steps:
      - uses: appleboy/ssh-action@v1
        with:
          host: ${{ secrets.VPS_HOST }}
          username: ${{ secrets.VPS_USER }}
          key: ${{ secrets.VPS_SSH_KEY }}
          port: ${{ secrets.VPS_PORT }}
          script: |
            cd /home/youruser/your-repo
            git pull origin main
            source venv/bin/activate
            pip install -r requirements.txt
            sudo systemctl restart fastapi-app

PART 11

Nginx Configurations

Different frameworks need different Nginx configs. Pick the one that matches your stack.

Reverse Proxy — Node.js / FastAPI / Django / Flask

/etc/nginx/sites-available/myapp
server {
    listen 80;
    server_name your_domain.com your_server_ip;

    location / {
        proxy_pass         http://localhost:3000;
        proxy_http_version 1.1;
        proxy_set_header   Upgrade $http_upgrade;
        proxy_set_header   Connection 'upgrade';
        proxy_set_header   Host $host;
        proxy_set_header   X-Real-IP $remote_addr;
        proxy_set_header   X-Forwarded-For $proxy_add_x_forwarded_for;
        proxy_cache_bypass $http_upgrade;
    }
}
✨
PORT MAP — Change 3000 to: 8000 (FastAPI/Django), 5000 (Flask), 3000 (Node.js/Next.js).

Static Site — React / Vue NEW

React and Vue build to static files. They don't need a proxy — Nginx serves them directly:

/etc/nginx/sites-available/my-react-app
server {
    listen 80;
    server_name your_domain.com;

    root /var/www/your-react-app;
    index index.html;

    # Handle client-side routing (React Router / Vue Router)
    location / {
        try_files $uri $uri/ /index.html;
    }

    # Cache static assets
    location ~* \.(js|css|png|jpg|jpeg|gif|ico|svg|woff2?)$ {
        expires 1y;
        add_header Cache-Control "public, immutable";
    }
}
⚠️
CRITICAL — The try_files directive is essential for single-page apps. Without it, refreshing any non-root URL (e.g. /about) returns a 404.

PHP-FPM — Laravel NEW

Laravel uses PHP-FPM, not a Node-style proxy:

/etc/nginx/sites-available/laravel-app
server {
    listen 80;
    server_name your_domain.com;

    root /home/youruser/your-repo/public;
    index index.php index.html;

    location / {
        try_files $uri $uri/ /index.php?$query_string;
    }

    location ~ \.php$ {
        fastcgi_pass unix:/run/php/php8.3-fpm.sock;
        fastcgi_param SCRIPT_FILENAME $realpath_root$fastcgi_script_name;
        include fastcgi_params;
    }

    location ~ /\.ht {
        deny all;
    }
}

Rollback Strategy NEW

When a deployment breaks production, you need to revert fast. Here's how to roll back for each stack.

Option 1: Git Revert (Safest)

Creates a new commit that undoes the last change. Keeps history clean and re-triggers CI/CD:

Your Local Machine
$ git revert HEAD
$ git push origin main    # Triggers CI/CD to deploy the revert

Option 2: Manual VPS Rollback (Fastest)

SSH into your VPS and checkout the previous working commit:

VPS Server
# See recent commits
$ git log --oneline -5

# Roll back to specific commit
$ git checkout <commit_hash> .

# Then restart your app
$ pm2 restart myapp              # Node.js
$ sudo systemctl restart myapp   # Python/PHP

Option 3: Database Rollback (Laravel/Django)

VPS Server
# Django — undo last migration
$ python manage.py migrate your_app <previous_migration_name>

# Laravel — rollback last batch
$ php artisan migrate:rollback
⚠️
ALWAYS BACKUP FIRST — Before any rollback involving database changes:
PostgreSQL: pg_dump yourapp_db > backup_$(date +%Y%m%d).sql
MySQL: mysqldump yourapp_db > backup_$(date +%Y%m%d).sql
TROUBLESHOOTING

Common Issues

❌
Permission denied (publickey) — Check ~/.ssh/authorized_keys and permissions (700 for .ssh/, 600 for authorized_keys). Also ensure the VPS has a key added to GitHub (Part 2 Step 4).
❌
Connection refused — SSH not running? sudo systemctl status sshd. Firewall? sudo ufw allow 22.
❌
502 Bad Gateway — App not running or wrong Nginx proxy_pass port. Check with: sudo systemctl status myapp or pm2 status.
❌
404 on page refresh (React/Vue) — Missing try_files $uri $uri/ /index.html; in Nginx config. See the Static Site config in Part 11.
❌
pm2 not found — Source nvm first: export NVM_DIR="$HOME/.nvm" && source "$NVM_DIR/nvm.sh"
❌
composer not found — Install: curl -sS https://getcomposer.org/installer | php && sudo mv composer.phar /usr/local/bin/composer
❌
Workflow not triggered — File must be at .github/workflows/deploy.yml. Check branch is main.
❌
git pull fails on VPS — VPS needs its own SSH key added to GitHub. See Part 2, Step 4. For HTTPS repos, switch to SSH: git remote set-url origin git@github.com:youruser/your-repo.git
❌
Build killed / out of memory — Add swap space (Part 5, Step 3). Check with free -h. For Node.js, also try: NODE_OPTIONS="--max-old-space-size=1024" npm run build
❌
Certbot fails — DNS must point to your server first. Verify: dig +short your_domain.com. Also ensure port 80 is open: sudo ufw allow 'Nginx Full'
❌
Laravel shows "500 Server Error" — Check permissions: sudo chown -R youruser:www-data storage bootstrap/cache and chmod -R 775 storage bootstrap/cache. Also check: php artisan config:clear
